## 一、产品概述

### 1.1 一句话定位

老舅厨房海外版（产品名暂定 **Precision Kitchen**）是一款面向家庭烘焙爱好者、咖啡冲煮玩家、特调饮品爱好者，以及家庭副业 / 微型工作室的**精准配方工作台**。核心差异化是把"量化引擎 + 受控 AI"结合，帮用户**做得稳、调得准、算得清**。

### 1.2 战略背景

- 国内微信小程序版（老舅厨房）已开发完成 14 个功能模块，主打中餐家庭厨房 SOP 与量化管理。
- 经市场判断：**中餐难以量化**（"少许、适量"是文化基因），而**烘焙 / 咖啡 / 饮品天生为量化而生**（baker's percentage、粉水比、糖度都是可精确表达的科学指标）。
- 因此海外版**砍掉正餐（一日三餐 / 餐单规划）逻辑**，聚焦三个垂直品类：烘焙、咖啡冲煮、特调饮品（含奶茶 / 鸡尾酒）。
- 国内版与海外版**共用同一套后端**，前端因平台（H5/小程序 vs Expo React Native）和品类差异需要重写。

### 1.3 市场依据（调研结论摘要）

| 结论 | 依据 |
|------|------|
| 奶茶 / 茶饮配方量化是市场空白 | 海内外搜索仅见奶茶店经营管理或消费打卡 app，无"家庭配方 + 精确比例 + 成本"产品 |
| 烘焙 / 咖啡 / 鸡尾酒是成熟红海 | 竞品密集（Bake-u-lator、BakerMath、Mixel、Brew Timer 等），需靠差异化切入 |
| 副业群体付费意愿强一个量级 | 家庭烘焙副业月入 500–2000 美元；BakeProfit 卖 $69/年仍被 1 万+ 用户采用 |
| 定价过低是副业者放弃首因 | 成本核算 + AI 定价直接命中此痛点 |
| 免费 app 占菜谱市场 73.6% 份额 | 免费层必须足够好用来获客，钱在副业付费线上 |

### 1.4 目标用户分层

```
免费层用户（增长引擎，占 95%+）
  → 纯爱好者，贡献流量、UGC、口碑，转化率 1–5%

PRO 付费用户（广度营收）
  → 进阶爱好者，愿为 AI 功能和无限配方付费

STUDIO 付费用户（利润 / 高 ARPU）
  → 家庭副业、微型工作室、市集摊主，为"赚钱工具"付费
```

## 四、核心功能详细设计

### 4.1 精确缩放引擎（改造重点）

#### 4.1.1 为什么要改

国内版缩放逻辑为中餐设计：`linear`（主料）/ `sub_linear`（调料 ^0.7）/ `fixed`（香料）。这套对中餐合理，但烘焙 / 饮品的缩放维度不同——不是"按食材类型分曲线"，而是"按比例基准联动"。

#### 4.1.2 引入 Scaling Profile（缩放模型）

每个配方绑定一种缩放模型：

| Scaling Profile | 基准 | 缩放逻辑 | 适用 |
|----------------|------|---------|------|
| `bakers_percentage` | 指定基准原料 = 100% | 其余按 % 联动；可锁定基准量或总重量反推 | 面包、蛋糕、面团、马卡龙 |
| `ratio_based` | 核心两元比例（如 1:15） | 锁定比例，一端变另一端联动 | 手冲咖啡、espresso |
| `multi_ratio` | **通用多组分比例**（泛化） | 多锚点，各自可锁定 | 奶茶、特调、**鸡尾酒** |
| `linear_legacy` | 份数 | 保留原线性逻辑 | 兼容旧配方 / 国内版 |

> **重要修正**：`multi_ratio` 从"奶茶专用"泛化为"通用多组分比例"，鸡尾酒（3 parts : 2 : 1）作为其特例。`bakers_percentage` 的基准原料**可指定任意原料**（不写死面粉），覆盖马卡龙（杏仁粉基准）、蛋白霜（蛋白基准）。

#### 4.1.3 数据模型改动

```
Recipe（配方）新增：
  scaling_profile  : enum  # bakers_percentage | ratio_based | multi_ratio | linear_legacy
  base_anchor      : jsonb # 基准锚点定义，随 profile 而变
  recipe_type      : enum  # simple | composite （MVP 只实现 simple，预留 composite）

RecipeIngredient（配方食材）新增：
  scaling_role     : enum  # anchor | percentage | ratio_linked | fixed
  percentage_value : decimal nullable   # baker's % 模式存百分比
  ratio_group      : string nullable    # multi_ratio 标记所属比例组
  scaling_correction : jsonb nullable   # 非线性修正规则（见 4.1.5）
```

`base_anchor` 示例：
```jsonc
// 烘焙
{ "anchor_ingredient_id": "flour_001", "anchor_is": "100%" }
// 手冲咖啡
{ "ratio": { "coffee": 1, "water": 15 }, "locked": "ratio" }
// 奶茶
{ "ratios": { "tea_to_water": "1:12", "sugar_pct": 8, "milk_pct": 30 } }
```

#### 4.1.4 缩放算法

**A. Baker's Percentage**
```
锁定基准量 F：  每原料量 = F × (percentage_value / 100)
锁定总重量 T：  S = Σ(percentage_value)
               F = T / (S / 100)
               再按上式算各原料
```

**B. Ratio-based（咖啡）**
```
比例 coffee:water = 1:R
调粉量 C → 水量 W = C × R
调水量 W → 粉量 C = W / R
（锁定端由 base_anchor.locked 决定）
```

**C. Multi-ratio（奶茶 / 鸡尾酒）**
```
多个独立锚点各自可锁定：
  茶汤量 = f(茶叶量, 茶水比)
  糖量   = 总液体量 × 糖度%
  奶量   = 总液体量 × 奶比%
调任一锚点，联动重算受影响项，未锁定项保持比例
```

#### 4.1.5 非线性修正（复用中餐 sub_linear 思想）

烘焙缩放有非线性陷阱：膨松剂和盐不线性缩放（缩放 >3 倍时泡打粉 / 小苏打约减 25%，盐放大 4 倍以上只用 3 倍量）。将国内版的 `sub_linear` 思想从"食材分类"重新定位为"缩放修正系数"：

```jsonc
scaling_correction: {
  "type": "step",
  "rules": [{ "above_factor": 3, "multiply": 0.75 }]
  // 缩放因子 > 3 时，该原料额外 × 0.75
}
```

修正规则做成"系统预设 + 用户可调"，AI 风味微调可在此给智能建议。

#### 4.1.6 边界与取舍

- **组合配方（千层 / 慕斯）**：单一 profile 套不进，需 `recipe_type = composite` 嵌套结构。**MVP 不实现，只预留字段**，V1.1+ 再做两层缩放。
- **冰淇淋 PAC/POD**：极专业小众，**主动排除**，`multi_ratio` 覆盖家用冰淇淋已足够。

### 4.3 AI 功能体系（核心差异化）

#### 4.3.0 统一设计铁律

> **AI 不"创造"内容，只在受控的知识库 / 规则范围内"匹配 + 排序 + 解释"，输出回写到量化引擎生成新版本。**

这是与市面"套壳 GPT 聊天 bartender"的根本区别——别人的 AI 是嘴，我们的 AI 是连着仪表盘的大脑。每个 AI 功能都遵循：
1. 输入用**结构化点选**（少打字，降弃用率）
2. AI 受**知识库 / 规则约束**（不发明）
3. 输出**回写引擎**生成新版本
4. 高风险动作有**幅度限制 + 预警**

#### 4.3.1 AI 配方解析导入【MVP】

- **场景**：用户拍照 / 粘贴杂乱配方文字 → AI 解析为结构化数据 → 进入缩放 / 成本引擎。
- **差异化**：竞品（如 Bake-u-lator）能扫描导入，但解析后无非线性缩放和成本引擎接住，我们有。
- **成本控制**：FREE 层限 5 次/月。

#### 4.3.2 AI 失败复盘诊断【V1.1，战略尖刀】★

**为什么能成立**：烘焙失败成因是收敛、可枚举的（蛋糕塌陷 → 膨松剂过量 / 烤温不对 / 中途开门 / 比例错误 / 搅拌过度），可建结构化因果知识库。

**三层漏斗架构（防错核心）**：
```
第一层 规则预筛（确定性，不用 AI）
  用配方数据硬性体检 → 输出候选成因列表（带初步权重）
  例：膨松剂 > 面粉 X% → 标记"过量嫌疑"
  例：实际烘焙时长 < 配方 75% → 标记"未熟透嫌疑"

第二层 AI 推理排序（LLM，受约束）
  输入候选成因 + 现象 + 环境 + 配方
  任务：在候选里排序 + 解释，禁止发明新成因
  强制结构化 JSON 输出，引用知识库条目 ID

第三层 修正回写引擎（确定性，不用 AI）
  把选定修正翻译为引擎参数变更 → 生成新配方版本 → 用户一键应用
```

**数据采集分四层**（大部分自动带入，最小化手填）：
| 数据层 | 来源 | 例子 |
|--------|------|------|
| 配方参数 | 自动（读版本） | 各原料克数、膨松剂比例、烤温、时长 |
| 执行偏差 | 半自动（计时器 / SOP 记录） | 实际时长、是否中途开门 |
| 环境变量 | 自动（联网天气）+ 手动 | 室温、湿度、海拔 |
| 失败现象 | 手动（结构化点选 + 照片） | 多选症状清单，非自由文本 |

**反过度自信机制**：证据不足时 AI 必须输出"需补充 X 数据"，禁止硬给高置信度结论。宁可少说，不可说错。

**知识库**（护城河）：`FailureSymptom`（现象）→ `FailureCause`（成因）→ `FixAction`（修正）多对多映射。MVP 靠**专家知识录入**打底（高度共识的烘焙原理），每品类先覆盖 10–15 种高频失败，不靠 AI 生成。

**MVP 范围**：只做一个品类（建议从蛋糕 / 面包切入，失败原因最结构化），验证后横向扩展。

#### 4.3.3 AI 风味微调【V1.1】

- **场景**：做完一版觉得"太甜 / 太苦 / 不够顺滑"，想调但不知动哪个参数。
- **笼子**：诉求结构化点选（滑杆 / 标签）；AI 只能输出现有原料的增减量，不能新增原料类别；输出回写引擎建新版本。
- **风险对冲**：单次单一原料调整幅度 ≤15%，提示小步迭代，做完记录反馈（回流复盘飞轮）。

#### 4.3.4 AI 定价助手【V1.1，副业付费钩子】

- **场景**：副业者定价，直击"定价过低是放弃首因"。
- **笼子**：输入成本引擎输出 + 工时 + 联网市场行情；输出**定价区间 + 依据拆解**（成本价 / 保本价 / 建议区间 / 利润率），非单一数字。
- **合规**：标注"仅供参考，非财务建议"；提醒美国多数州家庭食品作坊年收入上限（2.5 万–7.5 万美元，部分州低至 5000），帮算"会否撞上限"。

#### 4.3.5 AI 配料替代【V1.1】

- **场景**：缺料 / 过敏 / 无麸质 / 纯素。海外刚需（无麸质溢价 30–60%，客户极忠诚）。
- **笼子**：替代方案必须来自**替代知识库**（A → 可替代 B/C + 换算系数 + 风味影响），AI 不发明替代关系；输出自动重算配比 + 风味 / 质地预警。

### 4.4 多线程计时器冲煮化（改造，不重写）

国内版"最多 8 个并行计时器"内核完美适配，**做场景化包装**即可。

**三个改造点**：

1. **计时器升级为"带步骤的计时序列"**
   ```
   TimerSequence:
     steps: [{ name:"闷蒸", duration:30s, target:"注水 50g" },
             { name:"第一段", duration:45s, target:"注水至 150g" }, ...]
   与缩放引擎联动：粉量变 → 各段目标注水量按比例自动重算
   ```
   这是计时器与缩放引擎的联动点——别人的计时器是死的，我们的跟着配方比例走。

2. **SOP 暗色模式 → 冲煮 / 烘焙模式**
   复用现有暗色大屏，改内容为：大字当前步骤、倒计时、目标量、下一步预告。手冲场景尤其需要（手忙 / 湿，要一眼看懂）。

3. **检查点提醒（烘焙专用）**
   "前 25–30 分钟别开烤箱门，否则冷空气冲入塌陷"——支持节点提醒，直接预防失败，与复盘呼应。

### 4.5 成本核算 + 利润计算（保留 + 强化）【V1.1】

国内版"家庭省钱"，海外版"副业算利润"，付费动机强一个量级。价格内部存最小单位（如 $/g），展示按用户习惯单位转换。强化点：批量配方成本、利润率反算、与 AI 定价助手联动。

### 4.6 原料库改造（食材库 → 酒柜 / 豆仓 / 料台）

国内"我的食材库"改为按品类的原料库：烘焙料台、咖啡豆仓、酒柜。保留系统预设作搜索建议、库存为 0 标记"已耗尽"等既有逻辑。

### 5.2 定价结构：三层 + 一次性钩子

```
FREE（增长引擎）
  基础缩放、配方数上限、基础计时器、AI 解析导入（5 次/月）
  不含：成本 / 利润、AI 复盘、AI 定价、无限配方

PRO 爱好者层  $3.99/月 · $24.99/年
  无限配方、全缩放模型、多线程计时全功能、
  AI 风味微调、AI 复盘、AI 配料替代、冲煮/烘焙模式、配方分享

STUDIO 副业层  $8.99/月 · $79/年
  PRO 全部 + 成本核算 + 利润计算 + AI 定价助手
  + 订单/客户管理（未来）+ 批量成本

Lifetime PRO  $59 一次性（仅 PRO 层，抓反订阅用户；STUDIO 不提供买断）
```

**定价逻辑**：PRO $24.99 卡在竞品内容库（$12.99）与 BarGenius（$19.99）偏上，因功能更厚（AI + 引擎）；STUDIO $79 略高于 BakeProfit $69，副业用户多接一单即回本。

### 5.3 AI 成本红线（财务，非产品）

AI 是最大可变成本，定价必须覆盖：
- FREE：AI 解析限 5 次/月
- PRO/STUDIO：合理使用上限（如复盘 30 次/月），超出降速
- 后台监控每用户 AI 成本，确保 LTV > AI 成本 × 频次

### 5.4 上线节奏

```
MVP   ：只开 FREE + PRO（年费），验证"有人愿付 $24.99"
V1.1  ：上 AI 复盘 + 成本核算 → 开 STUDIO 层 + Lifetime PRO 钩子
```

### 5.5 必盯指标

免费转付费 1–5%。要么把免费池做大（增长），要么把转化做到行业上限（产品力）。STUDIO 副业层是拉高 ARPU 的关键。

| 风险 | 等级 | 对冲 |

## 九、给 Claude Code 的落地提示

1. **先核对实际代码**：本文档 `【待核对】` 项需对照 `server/src` 核实真实表名 / 字段名。
2. **后端改动最小化**：海外版优先复用现有 server，新增字段用 migration 增量加，不重构既有表。详见 `backend-reuse-guide.md`。
3. **前端重写**：mobile 用 Expo RN，参考 `frontend-architecture.md` 的目录结构与示例代码。
4. **feature flag 优先**：砍掉的功能（微信登录、餐单规划）用 flag 关闭，不删代码。
5. **小步提交**：按 RICE 顺序逐功能实现，每个功能独立可测、独立 PR。
